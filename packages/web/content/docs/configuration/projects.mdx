# Projects

The `projects` key is a map of project IDs to project configs. Each key is the project's identifier — used in session names, log paths, and the dashboard. Keys must match `[a-zA-Z0-9_-]+` (letters, numbers, hyphens, underscores — no dots or slashes).

```yaml
projects:
  my-app:          # project ID
    repo: org/my-app
    path: ~/my-app
```

## Required fields

| Field | Type | Description |
|-------|------|-------------|
| `repo` | `string` | GitHub/GitLab repository in `org/repo` format. Used by the SCM plugin to open PRs and check CI. |
| `path` | `string` | Absolute path to the local checkout. `~` is expanded to `$HOME`. |

## All project fields

### `name`

**Type:** `string` **Default:** project ID

Human-readable name shown in the dashboard. Defaults to the project key if not set.

```yaml
name: My App
```

### `defaultBranch`

**Type:** `string` **Default:** `"main"`

The branch agents target when opening PRs. Agents create feature branches off this branch.

```yaml
defaultBranch: main
```

### `sessionPrefix`

**Type:** `string` — must match `[a-zA-Z0-9_-]+` **Default:** derived from `path` basename

Short prefix used in tmux session names and worktree paths. AO auto-derives one from the directory name if not set. Set this explicitly when two projects would otherwise collide:

```yaml
sessionPrefix: app
```

### `agent`

**Type:** `string` **Default:** `defaults.agent` (built-in default: `claude-code`)

The AI agent to use for this project. Overrides `defaults.agent`.

```yaml
agent: codex
```

Available values: `claude-code`, `codex`, `aider`, `opencode`.

### `runtime`

**Type:** `string` **Default:** `defaults.runtime` (built-in default: `tmux`)

Where agent processes run. `tmux` gives you persistent sessions you can attach to from any terminal. `process` is lighter but sessions are lost on restart.

```yaml
runtime: tmux
```

### `workspace`

**Type:** `string` **Default:** `defaults.workspace` (built-in default: `worktree`)

How code is isolated per session. `worktree` is fast — it uses `git worktree add` on your existing clone. `clone` does a full `git clone`, which is slower but fully isolated.

```yaml
workspace: worktree
```

### `tracker`

**Type:** object **Default:** inferred from `repo`

Issue tracker configuration. AO infers `github` or `gitlab` from the repo URL if omitted.

```yaml
tracker:
  plugin: linear
  teamId: "your-linear-team-id"
```

For built-in trackers, specify `plugin`. For external trackers, specify `package` (npm) or `path` (local):

```yaml
tracker:
  package: "@acme/ao-plugin-tracker-jira"
  version: "^1.0.0"
```

Built-in tracker values: `github`, `linear`, `gitlab`.

#### trackerConfig — labels, assignee, milestone

Extra keys under `tracker` are passed to the tracker plugin as config. Common fields (GitHub tracker):

```yaml
tracker:
  plugin: github
  labels: [agent, automated]
  assignee: octocat
  milestone: "Q1 2025"
```

### `scm`

**Type:** object **Default:** inferred from `repo`

Source control management configuration. AO infers `github` or `gitlab` from the repo URL if omitted.

```yaml
scm:
  plugin: github
```

**Webhook acceleration (optional):** Instead of polling GitHub every 30 seconds, configure a webhook to receive instant CI and review events:

```yaml
scm:
  plugin: github
  webhook:
    path: /api/webhooks/github
    secretEnvVar: GITHUB_WEBHOOK_SECRET
    signatureHeader: x-hub-signature-256
    eventHeader: x-github-event
    deliveryHeader: x-github-delivery
    maxBodyBytes: 1048576
```

Set `GITHUB_WEBHOOK_SECRET` in your environment. Point the GitHub webhook at `http://your-host:3000/api/webhooks/github`.

### `symlinks`

**Type:** `string[]` **Default:** none

Files or directories to symlink from the project root into each workspace. Useful for `.env` files and agent config directories that agents need but should not be committed.

```yaml
symlinks:
  - .env
  - .claude
```

Paths are relative to `path`.

### `postCreate`

**Type:** `string[]` **Default:** none

Shell commands to run after a workspace is created but before the agent starts. Run in the workspace directory.

```yaml
postCreate:
  - "pnpm install"
  - "cp .env.example .env"
```

### `agentConfig`

**Type:** object **Default:** `{}`

Agent-specific configuration applied to all agents for this project.

#### `agentConfig.permissions`

**Type:** `"permissionless" | "default" | "auto-edit" | "suggest"` **Default:** `"permissionless"`

Controls how strictly the agent asks for permission before making changes.

| Value | Behavior |
|-------|----------|
| `permissionless` | Agent makes all edits without asking (also accepts the legacy alias `skip`) |
| `default` | Agent uses the tool's own default permission behavior |
| `auto-edit` | Agent auto-approves file edits but asks for other actions |
| `suggest` | Agent only suggests changes, never applies them |

```yaml
agentConfig:
  permissions: permissionless
```

#### `agentConfig.model`

**Type:** `string` **Default:** agent's built-in default

Override the model the agent uses. The accepted values depend on the agent:

```yaml
agentConfig:
  model: claude-sonnet-4-5
```

For Claude Code: `claude-opus-4`, `claude-sonnet-4-5`, `claude-haiku-3-5`, etc.
For Codex: `gpt-5-codex`, `o3`, etc.

#### `agentConfig.orchestratorModel`

**Type:** `string` **Default:** same as `model`

When using the decomposer or multi-agent mode, the model used by the orchestrator agent specifically.

### `orchestrator` and `worker`

**Type:** object **Default:** none

Per-role agent overrides. Useful when you want the orchestrator to use a more capable model and workers to use a faster one:

```yaml
orchestrator:
  agent: claude-code
  agentConfig:
    model: claude-opus-4

worker:
  agent: codex
  agentConfig:
    model: gpt-5-codex
```

### `agentRules`

**Type:** `string` **Default:** none

Inline instructions included in every agent prompt for this project. Written as a multi-line YAML string. This is the fastest way to add project-specific rules:

```yaml
agentRules: |
  Always run the test suite before pushing: `pnpm test`.
  Use conventional commits: feat:, fix:, chore:, etc.
  Never modify package.json directly — use pnpm add/remove.
  Target Node.js 20+. Do not use CommonJS require().
```

### `agentRulesFile`

**Type:** `string` **Default:** none

Path to a rules file relative to `path`. Use this when your rules are long or you want to version-control them separately:

```yaml
agentRulesFile: .agent-rules.md
```

If both `agentRules` and `agentRulesFile` are set, both are included in the prompt.

### `orchestratorRules`

**Type:** `string` **Default:** none

Rules applied only to the orchestrator agent (reserved for future multi-agent features). Same format as `agentRules`.

### `decomposer`

**Type:** object **Default:** disabled

Experimental issue decomposer. When enabled, AO uses a separate model to break down complex issues into sub-tasks before spawning agents.

```yaml
decomposer:
  enabled: true
  maxDepth: 3
  model: claude-sonnet-4-20250514
  requireApproval: true
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `enabled` | `boolean` | `false` | Enable the decomposer |
| `maxDepth` | `1–5` | `3` | Maximum decomposition depth |
| `model` | `string` | `claude-sonnet-4-20250514` | Model used for decomposition |
| `requireApproval` | `boolean` | `true` | Pause for human approval of the decomposition plan |

### `opencodeIssueSessionStrategy`

**Type:** `"reuse" | "delete" | "ignore"` **Default:** none

Only relevant when `agent: opencode`. Controls what AO does with an existing OpenCode session when re-spawning for the same issue:

| Value | Behavior |
|-------|----------|
| `reuse` | Attach to the existing session |
| `delete` | Delete the old session and create a fresh one |
| `ignore` | Ignore the old session and create a new one |

### `reactions` (per-project override)

**Type:** `Record<string, partial ReactionConfig>` **Default:** none

Override specific reaction settings for this project only. Keys match the global `reactions` keys. Only the fields you specify are overridden — unset fields fall through to the global reaction config.

```yaml
reactions:
  approved-and-green:
    auto: true      # enable auto-merge for this project
  ci-failed:
    retries: 5      # more retries for this project
```

See [Reactions](/docs/configuration/reactions) for the full list of reaction events and fields.

## Complete example

```yaml
projects:
  my-app:
    name: My App
    repo: org/my-app
    path: ~/projects/my-app
    defaultBranch: main
    sessionPrefix: app

    agent: claude-code
    runtime: tmux
    workspace: worktree

    tracker:
      plugin: linear
      teamId: "ENG"

    scm:
      plugin: github
      webhook:
        path: /api/webhooks/github
        secretEnvVar: GITHUB_WEBHOOK_SECRET

    symlinks: [.env, .claude]
    postCreate:
      - "pnpm install"

    agentConfig:
      permissions: permissionless
      model: claude-sonnet-4-5

    orchestrator:
      agentConfig:
        model: claude-opus-4

    agentRules: |
      Run `pnpm test` before every commit.
      Use conventional commits.

    agentRulesFile: .agent-rules.md

    reactions:
      approved-and-green:
        auto: true
```

## Common failure modes

**"Project ID must match `[a-zA-Z0-9_-]+`"** — your project key contains a dot or slash. Rename `org.my-app` to `my-app`.

**"Duplicate session prefix detected"** — two projects auto-derive the same prefix from their directory names. Add an explicit `sessionPrefix` to one of them.

**`symlinks` pointing at non-existent files** — AO logs a warning and skips the missing symlink. The workspace is still created.

**`postCreate` command fails** — AO logs the error and does not start the agent. Fix the command or check that dependencies are available.

**`tracker: plugin: linear` but no `teamId`** — the Linear plugin will throw at spawn time with "teamId is required". Add `teamId` under `tracker`.
