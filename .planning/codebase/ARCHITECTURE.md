# Agent Orchestrator Architecture

## System Pattern

- The repository is a pnpm monorepo defined in `pnpm-workspace.yaml`.
- The dominant pattern is a shared core plus adapters: `packages/core` defines contracts and orchestration logic, while `packages/cli`, `packages/web`, `packages/mobile`, and `packages/plugins/*` compose that core for different runtimes and user interfaces.
- `packages/core/src/types.ts` is the architectural spine. It defines the `Session` model and the plugin slots for runtime, agent, workspace, tracker, SCM, notifier, and terminal integrations.
- `packages/core/src/plugin-registry.ts` implements a slot-based registry keyed as `slot:name`. Built-in plugin names map to workspace packages under `packages/plugins/*`.
- The system is stateful but file-backed. Runtime state is persisted as flat metadata files through `packages/core/src/metadata.ts`, with paths and names generated by `packages/core/src/paths.ts`.

## Major Packages

- `packages/core/src/index.ts` exports the reusable platform: config loading, plugin registry, session manager, lifecycle manager, prompt building, decomposer, observability, feedback tools, and path utilities.
- `packages/cli/src/index.ts` is the main CLI entry point for the `ao` command and registers command modules from `packages/cli/src/commands/*`.
- `packages/agent-orchestrator/bin/ao.js` is the thin published wrapper package that points users to the CLI package.
- `packages/web/src/app/page.tsx` is the main dashboard page for the Next.js app.
- `packages/web/src/lib/services.ts` creates the web-side singleton for config, registry, session manager, and lifecycle manager.
- `packages/mobile/src/App.tsx` is a separate Expo client focused on mobile notifications and navigation over the same backend concepts.
- `packages/integration-tests/src/*.integration.test.ts` verifies end-to-end behaviors across runtime, agent, notifier, tracker, and dashboard flows.

## Entry Points

- CLI entry: `packages/cli/src/index.ts`.
- Published binary shim: `packages/agent-orchestrator/bin/ao.js`.
- Web UI entry: `packages/web/src/app/page.tsx`.
- Web API entry surface: `packages/web/src/app/api/**/route.ts`.
- Web terminal sidecars: `packages/web/server/terminal-websocket.ts` and `packages/web/server/direct-terminal-ws.ts`.
- Mobile app entry: `packages/mobile/src/App.tsx`.

## Data Flow

1. Config is loaded from `agent-orchestrator.yaml` by `packages/core/src/config.ts`.
2. Plugins are registered through `packages/core/src/plugin-registry.ts` or explicitly in `packages/web/src/lib/services.ts`.
3. Session creation flows through `packages/core/src/session-manager.ts`: workspace plugin creates isolated code space, agent plugin builds launch command and environment, runtime plugin starts execution, metadata is written to disk.
4. Runtime data is stored outside the repo using helpers in `packages/core/src/paths.ts`; the repo mostly stores code, templates, and docs.
5. Ongoing status polling runs through `packages/core/src/lifecycle-manager.ts`, which checks runtime liveness, agent activity, PR/CI/review state, and emits reactions.
6. Web and CLI surfaces call the same session manager and lifecycle manager abstractions instead of duplicating business logic.
7. Dashboard serialization and enrichment happen in `packages/web/src/lib/serialize.ts`, `packages/web/src/lib/services.ts`, and related lib modules before React components render.

## Server And Client Boundaries

- `packages/core` and most plugins are server-side Node code.
- `packages/cli` is fully server-side and talks directly to the local machine, tmux, git, trackers, and SCM providers.
- `packages/web` is split:
  - Server components and API routes live in `packages/web/src/app/**` and `packages/web/src/lib/**`.
  - Client components live in `packages/web/src/components/**` and interactive hooks under `packages/web/src/hooks/**`.
  - Terminal transport does not run inside Next route handlers; it runs in separate Node servers under `packages/web/server/**`.
- `packages/mobile` is a client application that depends on backend APIs and push notifications rather than direct local orchestration.

## Plugin Architecture

- Each plugin package follows the same shape: a `manifest` plus `create()` exported from `packages/plugins/<plugin-name>/src/index.ts`.
- Plugin slot coverage is visible in `packages/core/src/types.ts` and concretely loaded in `packages/core/src/plugin-registry.ts`.
- Examples:
  - Runtime plugin: `packages/plugins/runtime-tmux/src/index.ts`.
  - Agent plugin: `packages/plugins/agent-codex/src/index.ts`.
  - Workspace plugin: `packages/plugins/workspace-worktree/src/index.ts`.
  - Tracker plugin: `packages/plugins/tracker-linear/src/index.ts`.
  - SCM plugin: `packages/plugins/scm-github/src/index.ts`.
  - Notifier plugin: `packages/plugins/notifier-slack/src/index.ts`.
  - Terminal plugin: `packages/plugins/terminal-web/src/index.ts`.
- The plugin model is adapter-oriented rather than event-bus-oriented: core services call strongly typed interfaces defined in `packages/core/src/types.ts`.
- `packages/web/src/lib/services.ts` statically imports a subset of plugins because Next.js bundling cannot rely on dynamic `import()` resolution the same way the CLI/core can.

## Orchestration Loop

- Session spawning is owned by `packages/core/src/session-manager.ts`.
- Continuous state reconciliation is owned by `packages/core/src/lifecycle-manager.ts`.
- Status transitions map to event types, which then map to reaction configs and notifier behavior.
- Observability hooks live in `packages/core/src/observability.ts` and are used by both lifecycle logic and terminal servers.

## Practical Reading Order

- Start with `packages/core/src/types.ts`.
- Then read `packages/core/src/config.ts`, `packages/core/src/plugin-registry.ts`, and `packages/core/src/session-manager.ts`.
- Follow with `packages/core/src/lifecycle-manager.ts`.
- For dashboard behavior, continue into `packages/web/src/lib/services.ts`, `packages/web/src/app/api/sessions/route.ts`, and `packages/web/src/components/Dashboard.tsx`.
